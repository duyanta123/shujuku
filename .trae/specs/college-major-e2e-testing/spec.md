# 学院专业管理模块 E2E 自动化测试 Spec

## Why
学院专业管理系统已完成全栈开发，需要对管理员端页面交互、下拉级联、权限拦截、业务规则校验、边界异常场景进行全量 Playwright 自动化测试，确保所有功能稳定可靠，不回归破坏原有功能。

## What Changes
- **测试脚本**：使用 Playwright 编写 E2E 测试脚本，覆盖学院管理、专业管理、弹窗级联、权限拦截、业务规则、边界异常 6 大类场景
- **测试工具**：封装公共登录方法、页面跳转方法、截图工具，复用测试前置步骤
- **测试环境**：本地运行环境（前端 3000 / BFF 4000 / 后端 8080），使用独立测试账号
- **测试报告**：输出结构化测试报告，包含用例总数、通过数、失败数、失败详情与截图

## Impact
- Affected specs: `college-major-management-system`（测试覆盖该模块全部功能）
- Affected code: 无（纯测试脚本，不修改业务代码）
- New files: `tests/e2e/` 目录下所有测试文件

## ADDED Requirements

### Requirement: 权限与路由测试
系统 SHALL 验证管理员、学生、教师、未登录用户对学院专业管理模块的访问权限控制。

#### Scenario: 管理员可见学院专业管理菜单
- **WHEN** 管理员 admin/123456 登录
- **THEN** 侧边栏渲染「学院专业管理」菜单项，点击后正确跳转到 /admin/college-major 页面

#### Scenario: 学生侧边栏无学院专业管理菜单
- **WHEN** 学生 S001/123456 登录
- **THEN** 侧边栏不渲染「学院专业管理」菜单项，手动输入 URL /admin/college-major 自动跳转到学生首页

#### Scenario: 教师侧边栏无学院专业管理菜单
- **WHEN** 教师 T001/123456 登录
- **THEN** 侧边栏不渲染「学院专业管理」菜单项，手动输入 URL 自动跳转到教师首页

#### Scenario: 未登录用户访问管理页面
- **WHEN** 未登录状态下直接访问 /admin/college-major
- **THEN** 自动跳转到登录页面

### Requirement: 学院管理 Tab 功能测试
系统 SHALL 验证学院管理 Tab 的 CRUD 操作、表单校验、搜索筛选、删除校验等功能。

#### Scenario: 学院列表加载正常
- **WHEN** 管理员进入学院专业管理页面，默认显示学院管理 Tab
- **THEN** 学院列表正确展示，包含名称、状态（标签样式）、创建时间列

#### Scenario: 新增学院成功
- **WHEN** 管理员点击添加学院，输入合法名称，提交
- **THEN** 提交成功，列表自动刷新，新学院出现在列表中

#### Scenario: 新增学院名称唯一校验
- **WHEN** 管理员输入已存在的学院名称，提交
- **THEN** 后端返回错误提示「学院名称已存在」

#### Scenario: 新增学院表单校验
- **WHEN** 管理员提交空名称或超长名称（>100字符）
- **THEN** 前端表单校验拦截，显示对应错误提示

#### Scenario: 编辑学院名称
- **WHEN** 管理员点击编辑，修改学院名称，提交
- **THEN** 提交成功，列表数据同步更新

#### Scenario: 停用/启用学院
- **WHEN** 管理员停用学院后，再启用
- **THEN** 状态标签正确切换，停用学院在下拉选择器中不显示，启用后重新显示

#### Scenario: 删除无关联学院
- **WHEN** 管理员删除一个无关联专业、学生、教师的学院
- **THEN** 删除成功，列表移除该学院

#### Scenario: 删除有关联学院
- **WHEN** 管理员删除一个存在关联专业、学生或教师的学院
- **THEN** 删除失败，弹窗提示关联数据数量

#### Scenario: 学院列表搜索与筛选
- **WHEN** 管理员在搜索框输入学院名称关键字，或选择状态筛选条件
- **THEN** 列表正确过滤展示匹配结果

### Requirement: 专业管理 Tab 功能测试
系统 SHALL 验证专业管理 Tab 的 CRUD 操作、学院筛选、表单校验、删除校验等功能。

#### Scenario: 专业列表加载正常
- **WHEN** 管理员切换到专业管理 Tab
- **THEN** 专业列表正确展示名称、所属学院、状态列

#### Scenario: 按学院筛选专业
- **WHEN** 管理员在学院筛选下拉框中选择指定学院
- **THEN** 列表仅展示该学院下的专业

#### Scenario: 新增专业成功
- **WHEN** 管理员选择学院、输入合法专业名称，提交
- **THEN** 提交成功，列表刷新

#### Scenario: 同学院专业名称唯一校验
- **WHEN** 管理员在同一学院下创建同名专业
- **THEN** 后端返回错误提示「该学院下已存在同名专业」

#### Scenario: 不同学院同名专业允许
- **WHEN** 管理员在不同学院下创建同名专业
- **THEN** 提交成功

#### Scenario: 新增专业表单校验
- **WHEN** 管理员提交空名称、未选学院、超长名称
- **THEN** 前端表单校验拦截

#### Scenario: 编辑专业
- **WHEN** 管理员编辑专业，修改学院或名称，提交
- **THEN** 提交成功，数据同步更新

#### Scenario: 删除无关联专业
- **WHEN** 管理员删除无关联学生的专业
- **THEN** 删除成功

#### Scenario: 删除有关联专业
- **WHEN** 管理员删除存在关联学生的专业
- **THEN** 删除失败，提示关联学生数量

#### Scenario: 专业列表搜索与筛选
- **WHEN** 管理员搜索专业名称、按状态筛选
- **THEN** 列表正确过滤

### Requirement: 弹窗下拉级联测试
系统 SHALL 验证学生/教师/课程弹窗中学院-专业级联下拉的交互正确性。

#### Scenario: 学生弹窗学院下拉动态加载
- **WHEN** 管理员打开新增学生弹窗
- **THEN** 学院下拉框从 API 加载启用状态学院列表，支持 filterable 搜索过滤，有 loading 加载状态

#### Scenario: 学生弹窗专业级联
- **WHEN** 管理员在学生弹窗中选择学院
- **THEN** 专业下拉框自动加载对应学院的专业列表，切换学院自动清空已选专业

#### Scenario: 学生弹窗编辑回显
- **WHEN** 管理员编辑已有学生记录
- **THEN** 学院下拉回显原学院，专业下拉自动加载该学院专业列表并回显原专业

#### Scenario: 教师弹窗学院下拉
- **WHEN** 管理员打开新增/编辑教师弹窗
- **THEN** 学院下拉从 API 动态加载，必填校验生效，提交后数据正确保存

#### Scenario: 课程弹窗学院下拉与课程类型
- **WHEN** 管理员打开新增课程弹窗
- **THEN** 学院下拉从 API 加载，课程类型下拉可选必修/选修

#### Scenario: 编辑已选课课程禁用类型
- **WHEN** 管理员编辑已有学生选课记录的课程
- **THEN** 课程类型下拉框禁用，显示提示「该课程已有学生选课，无法修改课程类型」

### Requirement: 业务规则校验测试
系统 SHALL 验证必修课自动分配、跨学院选课拦截、教师绑课唯一性等核心业务规则。

#### Scenario: 学生新增自动分配必修课
- **WHEN** 管理员新增学生并指定有必修课配置的专业
- **THEN** 该学生自动获得对应专业的必修课选课记录

#### Scenario: 学生转专业重新分配必修课
- **WHEN** 管理员修改学生专业
- **THEN** 原专业必修课选课记录移除，新专业必修课自动分配，历史成绩考勤保留

#### Scenario: 必修课无法手动选课
- **WHEN** 学生通过选课接口尝试选择必修课
- **THEN** 后端拦截，返回「必修课无法手动选课」提示

#### Scenario: 跨学院选课拦截
- **WHEN** 学生尝试选择非本学院的选修课
- **THEN** 后端拦截，返回「仅可选择本学院选修课」提示

#### Scenario: 教师绑课唯一性
- **WHEN** 管理员尝试将已绑定课程的教师绑定到另一门课程
- **THEN** 后端拦截，返回「该教师已绑定其他课程」提示

#### Scenario: 教师调学院需解绑课程
- **WHEN** 管理员修改已绑定课程的教师所属学院
- **THEN** 后端拦截，返回「请先解除课程绑定」提示

#### Scenario: 专业仅能绑定本学院必修课
- **WHEN** 管理员尝试为专业绑定跨学院课程或选修课类型的课程
- **THEN** 后端拦截并返回对应错误提示

### Requirement: 交互体验与边界测试
系统 SHALL 验证表单防重复提交、空数据状态、网络异常处理等交互体验。

#### Scenario: 表单提交按钮防重复
- **WHEN** 管理员提交表单
- **THEN** 提交按钮进入 loading 禁用态，请求完成后恢复

#### Scenario: 空数据状态展示
- **WHEN** 列表无数据
- **THEN** 展示空状态提示，不出现异常

#### Scenario: 页面刷新后 Tab 状态保留
- **WHEN** 管理员在专业管理 Tab 刷新页面
- **THEN** Tab 状态与筛选条件合理保留或重置
